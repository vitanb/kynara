Security & Trust

Kynara is the control plane that decides what your AI agents can do — so we hold ourselves to the security bar our customers expect of a control plane. Here's how Kynara is built.

Architecture

Outside the LLM trust boundary

Kynara evaluates structured requests — subject, action, resource, context — not natural language. A prompt injection inside the model cannot change what Kynara receives or how it decides.

Fail-closed by default

If the control plane is unreachable, the SDKs deny by default (fail_closed=True). An agent loses access on failure rather than gaining it.

Non-escalation guarantee

An agent acting on behalf of a user can never exceed that user's permissions — the effective set is the intersection of both.

Least privilege

Deny-by-default policies, scoped capabilities, and least-privilege tool discovery shrink the blast radius of any single agent.

Authentication & secrets

  • Password hashing with Argon2id; human sessions use short-lived JWTs with rotating refresh tokens.
  • Scoped API keys for agents and machines — each key declares exactly the scopes it may use (e.g. decisions.check).
  • Secrets encrypted at rest (AES via Fernet) — upstream tokens, integration credentials, and identity-provider tokens are never stored in plaintext.
  • Enterprise SSO via SAML 2.0 and OIDC, with SCIM provisioning.

Data & tenancy

  • Multi-tenant isolation with row-level security in Postgres; every record is scoped to an organization.
  • Data residency & BYOK options for regulated workloads.
  • Self-host — Kynara is source-available (BSL 1.1), so you can run the full stack in your own environment and inspect every line. SDKs are MIT.

Tamper-evident audit

Every decision — allow, deny, or require_approval — is appended to a SHA-256 hash-chained, append-only log. Each record links to the previous one, so any modification or deletion of a past entry breaks the chain and is detectable on the next integrity check. Logs are queryable and exportable, supporting incident reconstruction and audit evidence (including EU AI Act Article 12 logging expectations). See our guide on agent logging.

Compliance posture

Kynara's controls are designed to support your SOC 2, ISO 27001, GDPR, and HIPAA programs — tamper-evident audit, least privilege, encryption at rest, access reviews, and approval workflows map directly to common framework requirements. Enterprise plans include a HIPAA BAA and custom retention. For our current attestation status and to request security documentation, contact us.

For AI-agent-specific controls, see how Kynara maps to the OWASP AI Exchange and MITRE ATLAS mitigations — least model privilege, oversight, and dynamic downgrade on untrusted input.

We maintain a threat model and conduct security testing of the platform. Responsible-disclosure reports are welcome — see our disclosure policy or email security@kynaraai.com.
