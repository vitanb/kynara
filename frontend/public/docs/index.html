Kynara Documentation

Complete reference for deploying, configuring, and operating Kynara — the AI agent permission and governance platform.

Overview

Kynara is an AI agent permission control plane. It intercepts every tool call your agents make, evaluates it against your organisation’s policies, and returns one of three outcomes: allow, deny, or require_approval. Every decision is logged to a tamper-evident audit chain.

What Kynara does

  • Enforces policies at runtime — before an agent touches a resource, not after.
  • Captures a full audit trail — every decision, approval, and access event with chain-hash integrity.
  • Manages the full agent identity lifecycle — credential issuance, rotation, revocation, and delegation.
  • Provides human-in-the-loop approval workflows — with Slack, Teams, and PagerDuty notifications.
  • Exports compliance evidence — one-click SOC 2 ZIP, HIPAA access CSV, quarterly access review.

Architecture

Kynara is deployed in one of four tiers, from least to most invasive:

TierIntegration pointLatency addedBest for
0Transparent HTTP reverse proxy (port 8080)~2–5 msAny HTTP-based LLM API call
1MCP server wrapper (port 9090)~1–3 msAgents using the MCP protocol
2Go policy sidecar (in-process, <1 ms)<1 msHigh-throughput production agents
3Python / TypeScript SDK decorator<1 msDirect SDK integration

Quickstart

Get an agent governed in under 5 minutes using Tier 0 (no code changes required).

1. Get your API key

Log in to kynaraai.com → Settings → API Keys → Create key. Copy the key — it is shown once.

2. Start the proxy

docker run -p 8080:8080 \
  -e KYNARA_API_KEY=kyn_your_key \
  -e KYNARA_AGENT_ID=my-first-agent \
  -e UPSTREAM_URL=https://api.openai.com \
  ghcr.io/kynara/kynara-proxy:latest

3. Point your agent at the proxy

from openai import OpenAI
client = OpenAI(base_url="http://localhost:8080/v1", api_key="any")
# All calls now flow through Kynara

4. Create your first policy

In the dashboard: Policies → New policy. Set effect to require_approval for email:send actions. Your agent will now pause and wait for a human before sending any email.

Core concepts

ConceptDescription
AgentA registered AI process with a unique ID, mode (autonomous / human_supervised / read_only), and a daily action budget.
PolicyA rule with an effect (allow / deny / require_approval), a list of actions it matches, and an optional condition expression.
DecisionThe result of evaluating a request against all policies. Logged immutably to the audit chain.
ApprovalA pending human review created when a decision returns require_approval. The agent blocks until resolved.
JIT grantA time-bound scope elevation (5–480 min). Used for break-glass access with mandatory justification.
ScopeA permission string like crm:contacts.read. Roles are sets of scopes. Agents inherit scopes via role assignments.
BundleA signed JSON snapshot of all policies for an org. The Go sidecar fetches and caches the bundle for sub-millisecond local evaluation.

Deployment tiers

Choose the tier that best fits your integration constraints. You can mix tiers — e.g., Tier 0 for legacy agents and Tier 3 for new ones in the same org.

Tier 0 — Transparent HTTP Proxy

A reverse proxy that intercepts every outbound HTTP request your agent makes. No code changes required — just point the agent’s base URL at the proxy.

Docker Compose

version: "3.9"
services:
  kynara-proxy:
    image: ghcr.io/kynara/kynara-proxy:latest
    ports: ["8080:8080"]
    environment:
      KYNARA_API_KEY: kyn_your_key
      KYNARA_AGENT_ID: agent-001
      UPSTREAM_URL: https://api.openai.com
      KYNARA_API_URL: https://api.kynaraai.com
    restart: unless-stopped

How it works

  1. Agent sends POST /v1/chat/completions to proxy (port 8080).
  2. Proxy extracts the tool name and arguments from the request body.
  3. Proxy calls POST /api/v1/decisions on the Kynara backend.
  4. If allow → request is forwarded upstream. If deny → 403 returned. If require_approval → 202 with approval_url.

Decision request (sent internally by proxy)

POST https://api.kynaraai.com/api/v1/decisions
Authorization: Bearer kyn_your_key
Content-Type: application/json

{
  "agent_id": "agent-001",
  "action": "tool:send_email",
  "resource": {"type": "email", "to": "ceo@example.com"},
  "context": {"tier": "0", "model": "gpt-4o"}
}

Decision response

{
  "effect": "require_approval",
  "decision_id": "dec_01HXYZ",
  "approval_id": "apr_01HABC",
  "approval_url": "https://kynaraai.com/approvals/apr_01HABC",
  "reason": "Policy: require-approval-for-email-sends",
  "latency_ms": 3
}

Tier 2 — Go Policy Sidecar

A Go binary that runs as a sidecar alongside your agent process. It fetches the policy bundle from Kynara on startup, caches it locally, and evaluates decisions in <1 ms without a network hop.

How the bundle works

  1. Sidecar calls GET /api/v1/policy-bundle/sidecar with the org API key.
  2. Backend generates a signed bundle (Ed25519) containing all enabled policies.
  3. Sidecar verifies the signature using GET /api/v1/policy-bundle/pubkey.
  4. Decisions are evaluated locally from the cached bundle. Audit events are streamed back to the backend in batches.
  5. Sidecar polls for bundle updates every 30 seconds, or immediately on a bundle.updated webhook.

Bundle signing

Every bundle is signed with the org’s Ed25519 private key (stored KMS-encrypted). The sidecar rejects any bundle with an invalid or missing signature.

GET /api/v1/policy-bundle/pubkey
Authorization: Bearer kyn_your_key

# Response
{
  "org_id": "org_01HX",
  "public_key_pem": "-----BEGIN PUBLIC KEY-----\nMCowBQ...\n-----END PUBLIC KEY-----\n"
}

# Rotate the signing keypair (sidecar must re-fetch bundle after rotation)
POST /api/v1/policy-bundle/rotate-signing-key

Framework Integrations

Drop-in integrations for every major agent framework. All follow the same pattern: intercept at the framework’s tool-dispatch boundary so your tool implementations stay unchanged.

LangGraph

from kynara_sdk import Kynara, PermissionDenied, ApprovalRequired
from langgraph.prebuilt import ToolNode

class KynaraToolNode(ToolNode):
    """Drop-in ToolNode replacement that gates every call through Kynara."""
    def _run_one(self, call, config):
        try:
            kynara.enforce(
                subject=("agent", AGENT_ID),
                action=f"tool:{call['name']}",
                resource={"tool": call["name"], "args": call.get("args", {})},
            )
        except PermissionDenied as e:
            from langchain_core.messages import ToolMessage
            return ToolMessage(content=f"[DENIED] {e.decision.reason}", tool_call_id=call["id"])
        return super()._run_one(call, config)

# Use in your graph
graph.add_node("tools", KynaraToolNode(tools))

LlamaIndex

from kynara_sdk import Kynara
from llama_index.core.tools import FunctionTool
import functools

def kynara_tool(fn, action, **kwargs):
    @functools.wraps(fn)
    def guarded(**kw):
        kynara.enforce(subject=("agent", AGENT_ID), action=action, resource=kw)
        return fn(**kw)
    return FunctionTool.from_defaults(fn=guarded, **kwargs)

read_tool = kynara_tool(read_file, action="file:read", name="read_file")

Agent-to-Agent Delegation

A parent agent can delegate a subset of its own scopes to a child agent for a limited time. The delegated scopes cannot exceed the parent’s own scopes (no privilege escalation). Delegation chains are depth-limited.

  "max_chain_depth": 1,
  "justification": "Sub-agent needs read access for this sprint",

}

GET /api/v1/agents/{agent_id}/received-delegations  # delegations TO this agent

Agent Risk Scoring

Kynara computes a rolling risk score (0–100) for each agent daily, based on its decision history over the last 30 days. Agents above threshold are auto-escalated to a stricter risk class.

Score components

FactorWeightDescription
Deny rate40%% of decisions that were denied
Approval rate30%% of decisions requiring human approval
Anomaly flag rate30%% of decisions flagged by anomaly detector

Risk classes

ScoreClassEffect
0–74low / mediumNormal operation
75–89highElevated monitoring; approval thresholds lowered
90–100criticalAll non-read actions require approval

Policies

Policies define what agents are allowed to do. Each policy matches a set of actions and returns an effect.

Create a policy

POST /api/v1/policies
{
  "slug": "block-prod-deletes",
  "display_name": "Block production deletes",
  "effect": "deny",
  "priority": 100,
  "actions": ["database:delete", "storage:delete"],
  "resource_types": ["production"],
  "condition": {"environment": "production"},
  "is_enabled": true
}

Effects

EffectBehaviour
allowAction proceeds immediately.
denyAction blocked. HTTP 403 returned to agent.
require_approvalAction paused. HTTP 202 + approval_url returned. Agent polls until resolved.

Priority

Lower number = higher priority. A deny at priority 100 beats an allow at priority 400. Policies are evaluated in ascending priority order and the first match wins.

Downloads a ZIP containing: full audit log CSV, chain-integrity verification JSON, access review CSV, and decision summary JSON. Suitable as evidence for a SOC 2 Type II audit.

GET /api/v1/audit/compliance/soc2?since=2026-02-01T00:00:00Z&until=2026-05-01T00:00:00Z
# Returns: soc2-evidence-20260201-20260501.zip

All access events (auth, data read/write, policy decisions) for the period. Required for HIPAA audit controls §164.312(b).

GET /api/v1/audit/compliance/hipaa?since=2025-06-01T00:00:00Z
# Returns: hipaa-access-20250601-20260530.csv

Current org members with roles, last-login timestamps, and any active JIT grants. Designed for quarterly access review sign-off.

Bundle Signing (Ed25519)

Policy bundles fetched by the Go sidecar are signed with an org-specific Ed25519 private key stored in KMS. The sidecar verifies the signature before loading any bundle, preventing man-in-the-middle tampering.

# Fetch the signing public key
GET /api/v1/policy-bundle/pubkey

# Rotate keys (invalidates all cached bundles)
POST /api/v1/policy-bundle/rotate-signing-key

# Verify a bundle locally (Python)
from app.security.bundle_signing import verify_bundle
is_valid = verify_bundle(bundle_dict, public_key_pem_bytes)

Data Residency

Enterprise orgs can pin their data to a specific deployment region. If a request reaches the wrong region, Kynara returns HTTP 451 (Unavailable For Legal Reasons).

EU Central (Frankfurt)api-euc1.kynaraai.com

Enable strict residency in org settings. With residency_strict=true, any request reaching the wrong region is rejected before any data is processed.

SIEM Integration

Forward audit events to your SIEM in real time using one of three methods:

  1. Webhook — Kynara POSTs events to your SIEM ingest URL as JSON.
  2. Polling — Call GET /api/v1/audit/events?since_sequence=N from your collector on a schedule.
  3. SSE stream — Subscribe to GET /api/v1/activity/stream for real-time push delivery.

Example: Splunk HEC via webhook

POST /api/v1/webhooks
{
  "url": "https://splunk.example.com:8088/services/collector",
  "secret": "splunk-hec-token",
  "events": ["*"]
}

If a payment fails, the subscription enters past_due status. Agents continue operating normally for a 7-day grace period. After 7 days, all agent actions are blocked until payment is resolved.

FAQ

What happens if the Kynara API is unreachable?
Tier 0/1 proxies fail-open or fail-closed depending on your KYNARA_FAIL_POLICY setting (open / closed, default closed). The Go sidecar (Tier 2) uses its cached bundle for up to 5 minutes before failing closed.

How long does a policy decision take?
Tier 0/1 add 2–5 ms. The Go sidecar (Tier 2) evaluates in <1 ms locally.

Can I use Kynara without changing any agent code?
Yes — use Tier 0 (HTTP proxy) or Tier 1 (MCP wrapper). No code changes required.

How do I manage policies without the dashboard?
Use the CLI: python scripts/kynara-cli.py pull / push / diff. Or connect a Git repository via the Git Sync feature for GitOps-style management.

Can agents request their own temporary access?
Human users (not agents themselves) can submit self-service JIT elevation requests via POST /api/v1/jit-grants/request. An admin approves, and the grant is auto-created.

How is the audit log protected against tampering?
Each entry includes a SHA-256 hash of the previous entry. Any modification breaks the chain and is detected by POST /api/v1/audit/verify.

What frameworks are supported?
LangChain, LangGraph, LlamaIndex, CrewAI, AutoGen, OpenAI Assistants, AWS Bedrock Agents, Google Vertex AI, Microsoft Semantic Kernel. See the Framework integrations section.
