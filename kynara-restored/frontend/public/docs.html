Kynara User Guide

Everything you need to register agents, define access policies, and audit every decision made by your AI agents — in real time.

Overview

Kynara is an AI agent permission control plane. It sits between your AI agents and the resources they act on, evaluating every tool call against your organisation's policies before allowing it to proceed.

🛡️ Policy enforcement

ABAC rules evaluated in priority order. First matching decision wins; the default is deny.

📋 Full audit trail

Every allow, deny, and approval request is logged with actor, context, and outcome.

⚡ Sub-5ms decisions

Cached policy evaluation so your agents never slow down waiting for permission checks.

👥 Human-in-the-loop

Policies can require an approval step before sensitive actions are executed.

Architecture at a glance

Your agent sends a decision request to the Kynara API, describing the action it wants to take (action), the thing it's acting on (resource), and who it's acting for (principal). Kynara evaluates your policies, returns allow, deny, or require_approval, and appends an immutable record to the audit log. 

Agent  →  POST /api/v1/decisions/check  →  Kynara
                                              ↓
                              Policy engine evaluates rules
                                              ↓
                  ←  { outcome: "allow" | "deny" | "require_approval" }
                                              ↓
                                    Audit event appended

Quick start

Get from zero to your first policy decision in under five minutes.

  1. Create your account at kynara.ai. The free plan includes 3 seats and 10,000 policy decisions per month.

  2. Register your first agent. Navigate to Agents → New agent. Give it a display name, choose a supervision mode (full-auto, supervised, or manual), and optionally set rate limits. Copy the Agent ID and API key — you'll need these in your code.

  3. Create a policy. Navigate to Policies → New policy. For example, create an Allow read-only file access policy with actions: ["file.read"], resources: ["file"], and effect: allow.

  4. Bind the policy to your agent. Open the agent detail page and click Attach policy to link the policy you just created.

  5. Call the decision API from your agent:

curl -X POST https://kynara.ai/api/v1/decisions/check \
  -H "Authorization: Bearer <your-api-key>" \
  -H "Content-Type: application/json" \
  -d '{
    "agent_id": "<agent-uuid>",
    "action": "file.read",
    "resource_type": "file",
    "resource_id": "reports/q4.csv",
    "context": { "user_id": "u_123" }
  }'

A response of {"outcome": "allow"} means your policy matched and the action is approved. You'll see the decision appear in the Audit log immediately.

💡
No policy = deny. If no policy matches a request, Kynara returns deny by default. Start with broad allow policies and narrow them down — not the other way around.

Key concepts

Agents

An agent is a registered identity for an AI process. Every agent has a unique ID, an API key, a supervision mode, and an optional rate limit. Agents are the subject of policy evaluation.

Policies

A policy is an ABAC rule that defines what an agent can do. Each policy has a priority, a set of actions, a set of resource types, conditions (optional), and an effect: allow, deny, or require_approval.

Policies are evaluated in ascending priority order. The first policy whose conditions match the request wins. If no policy matches, the default decision is deny.

Policy bindings

A policy binding attaches a policy to a subject. A binding can target a specific agent (agent:<uuid>) or all agents in the org (*). Org-wide bindings are useful for baseline rules like "deny all shell executions".

Decision outcomes

OutcomeMeaning
allowThe action is permitted. The agent may proceed.
denyThe action is blocked. The agent must not proceed.
require_approvalA human must approve before the action can proceed. The agent should pause and wait.

Supervision modes

ModeBehaviour
full_autoDecisions are evaluated against policies with no additional human checkpoint.
supervisedHigh-sensitivity actions escalate to a human reviewer regardless of policy.
manualEvery action requires explicit human approval before execution.

Tools

Tools are the callable functions your agents expose. Registering a tool in Kynara lets you reference it precisely in policies (e.g., action: "tool.execute", resource_type: "tool", resource_id: "send_email").

Getting started guide

If your organisation has no agents registered yet, the dashboard displays a three-step guide to help you get up and running quickly.

Agents

Creating an agent

  1. Navigate to Agents → New agent.

  2. Enter a display name (e.g., email-drafter-prod) and an optional description.

  3. Choose a supervision mode: full_auto, supervised, or manual.

  4. Set an optional rate limit (requests per minute). Leave empty for unlimited.

  5. Click Create agent. You will be taken to the agent detail page where the API key is shown once — copy it immediately.

⚠️
API keys are shown only once. Store the key in a secret manager (AWS Secrets Manager, HashiCorp Vault, etc.) before navigating away.

Agent detail page

The agent detail page shows identity information, the current supervision mode, rate limits, and the list of policies bound to this agent. You can:

  • Attach a new policy via the Attach policy button.
  • Remove a policy binding by clicking the trash icon next to it.
  • Deactivate the agent with the Deactivate toggle.
  • Rotate the API key with Rotate key.

Policy bindings on an agent

Each policy shown on the agent detail page displays whether it was bound directly (Direct label) or via an org-wide wildcard binding (Org-wide label). Org-wide bindings cannot be removed from the agent detail page — manage them from the Policies page instead.

Deactivating vs deleting an agent

Deactivating an agent sets is_active: false. All decision requests from that agent will receive deny responses. Audit history is preserved. Deleting an agent is permanent and removes all bindings; audit history is retained for compliance purposes.

Policies

Creating a policy

  1. Navigate to Policies → New policy.

  2. Enter a display name and optional description.

  3. Set a priority (lower number = evaluated first). Use gaps (e.g., 100, 200, 300) so you can insert policies later without renumbering.

  4. Specify actions as a comma-separated list (e.g., file.read, file.write). Leave empty or use * to match all actions.

  5. Specify resource types (e.g., file, database). Leave empty to match all resource types.

  6. Choose an effect: allow, deny, or require_approval.

  7. Toggle Enabled on and click Save.

Policy evaluation order

When Kynara receives a decision request, it evaluates all enabled policies bound to the requesting agent (directly or via *) in ascending priority order. The first match wins. If no policy matches, the outcome is deny.

💡
Tip: Place broad deny rules at low priority numbers and narrower allow rules at higher priority numbers to create an allow-list model. Or do the reverse for a deny-list model — it depends on your security posture.

Binding policies to agents

A policy has no effect until it is bound to at least one subject. You can bind from two places:

  • Agent detail page → Attach policy: creates an agent:<uuid> binding.
  • Policy detail page → Add binding: lets you choose a specific agent or all agents (*).

Disabling a policy

Setting a policy to disabled excludes it from evaluation without removing its bindings. This is useful for temporarily suspending a rule during an incident without losing configuration.

Audit log

The audit log records every event that passes through Kynara. It is append-only and SHA-256 hash-chained, making tampering detectable.

Event types

Event typeWhen it fires
policy.decisionEvery time the decision engine evaluates a request.
agent.createdA new agent is registered.
agent.deactivatedAn agent's active flag is set to false.
policy.createdA new policy is saved.
policy.updatedAn existing policy is modified.
auth.loginA user signs in (email/password or SSO).
auth.logoutA user signs out.
member.invitedA team invitation is sent.

Filtering

Use the filter controls at the top of the Audit log page to narrow by event type, agent, outcome, or date range. Filters are applied server-side.

Exporting

Click Export CSV to download all matching events as a CSV file. Enterprise plans support streaming exports to S3-compatible storage.

✅
Compliance note: The audit log is designed to satisfy SOC 2 Type II log-integrity requirements. The hash chain means that any deletion or modification of a log record will break the chain and be detected on the next integrity check.

Tool registry

The tool registry lets you define the callable functions your agents expose so that you can reference them precisely in policies.

Registering a tool

  1. Navigate to Tools → New tool.

  2. Enter a name (this becomes the resource_id in decision requests).

  3. Add an optional description and schema (JSON Schema for the tool's inputs).

  4. Click Save.

Once registered, you can write policies that target a specific tool: action: "tool.execute", resource_type: "tool", resource_id: "send_email".

Decision SDK

Rather than calling the HTTP API directly, you can use one of the official SDK wrappers. (SDKs are in beta — see the GitHub repo for the latest status.)

Python

pip install kynara

from kynara import KynaraClient

client = KynaraClient(api_key="sk_...")

result = client.check(
    agent_id="<uuid>",
    action="file.read",
    resource_type="file",
    resource_id="reports/q4.csv",
)

if result.outcome == "allow":
    # proceed
elif result.outcome == "require_approval":
    # pause and poll for approval
else:
    raise PermissionError("Action denied by policy")

Node.js / TypeScript

npm install @kynara/sdk

import { KynaraClient } from "@kynara/sdk";

const client = new KynaraClient({ apiKey: "sk_..." });

const { outcome } = await client.check({
  agentId: "<uuid>",
  action: "file.read",
  resourceType: "file",
  resourceId: "reports/q4.csv",
});

if (outcome !== "allow") throw new Error("Policy denied");

SSO / Okta

Pro and Enterprise plans support SAML 2.0 and OIDC SSO via Okta (and other compatible providers).

Configuring Okta

  1. In your Okta admin console, create a new OIDC application. Set the redirect URI to https://kynara.ai/api/v1/auth/sso/okta/callback.

  2. Copy the Client ID, Client Secret, and Okta domain.

  3. In Kynara, navigate to Settings → SSO.

  4. Enter the Okta domain, Client ID, and Client Secret. Click Save.

  5. Test the integration by clicking Test SSO connection.

Once configured, users can click Continue with Okta on the sign-in page to authenticate via SSO. JIT (just-in-time) provisioning creates a Kynara account on first login.

⚠️
Do not disable password login until you have verified at least one owner can sign in via SSO. If SSO misconfiguration locks you out, contact support.

Policy schema reference

{
  "display_name": "Allow read-only file access",
  "description": "Agents may read any file. Write operations are not covered.",
  "priority": 100,
  "effect": "allow",               // "allow" | "deny" | "require_approval"
  "actions": ["file.read"],        // [] or ["*"] = match all
  "resource_types": ["file"],      // [] or ["*"] = match all
  "conditions": {                  // optional, ABAC attribute conditions
    "context.environment": "production"
  },
  "is_enabled": true
}

Field reference

FieldTypeRequiredDescription
display_namestringYesHuman-readable name shown in the UI.
descriptionstringNoLonger explanation of the policy's intent.
priorityintegerYesEvaluation order. Lower = evaluated first.
effectenumYesallow, deny, or require_approval.
actionsstring[]NoAction strings to match. Empty or ["*"] = all actions.
resource_typesstring[]NoResource type strings to match. Empty = all types.
conditionsobjectNoKey-value pairs matched against the request context.
is_enabledbooleanNoDefaults to true. Disabled policies are skipped.

Decision outcomes reference

allow

A matching policy with effect: allow was found. The requesting agent is cleared to proceed. The decision is logged with the matching policy ID.

deny

Either a matching policy with effect: deny was found, or no policy matched at all (default-deny). The agent must not proceed. Always log denials on the agent side for debugging.

require_approval

A matching policy with effect: require_approval was found. The agent should pause execution and wait for a human to approve or reject the request in the Kynara UI (Audit log → Approvals). A webhook notification is sent to your configured endpoint when a pending approval is resolved.

FAQ

What happens if the Kynara API is unreachable?

Kynara is fail-closed. If your agent cannot reach the API, treat the response as deny. Never default to allow on timeout. The SDK wrappers enforce this behaviour automatically.

Can I have policies that apply to all agents?

Yes. Create a policy binding with subject_selector: "*". This is useful for org-wide deny rules (e.g., "no agent may execute shell commands").

How long is audit log data retained?

Free plans retain 30 days of audit log data. Pro plans retain 1 year. Enterprise plans support custom retention and export to your own storage.

Is my data encrypted at rest?

Yes. All data is encrypted at rest (AES-256) and in transit (TLS 1.3). Enterprise plans support customer-managed encryption keys (CMEK).

Can I import policies from code (GitOps)?

Yes. Use the REST API (POST /api/v1/policies) from your CI pipeline. A Terraform provider is on the roadmap for IaC-native policy management.

How do I rotate an agent API key?

Open the agent detail page and click Rotate key. The old key is immediately invalidated. Copy the new key before closing the modal.

What regions is Kynara available in?

Currently us-east-1. EU (eu-west-1) and APAC regions are on the roadmap. Enterprise plans can request a dedicated single-tenant deployment.
